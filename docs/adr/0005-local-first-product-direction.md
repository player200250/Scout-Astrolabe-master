# ADR 0005：本地優先（Local-First）的產品方向

## 狀態

已採用（截至 2026-05-08）；**2026-06-21 經 [ADR 0006](0006-cloud-sync-and-mobile.md) 修正**——新增「可選的雲端同步層 + 行動端 PWA」，本機核心原則維持不變（IndexedDB 仍為 ground truth、同步為可選附加層）。

---

## 背景

Scout Astrolabe 在設計時面臨一個根本性的架構選擇：資料存放在哪裡、如何在多裝置間同步？

現代筆記/白板工具通常採用兩種模式：

1. **雲端優先（Cloud-first）**：資料存雲端，本機為快取；需要帳號、訂閱制、網路依賴
2. **本地優先（Local-first）**：資料存本機，使用者完全擁有資料；可選擇性加入同步

Scout Astrolabe 的目標使用者是重視**資料所有權**和**離線可用性**的知識工作者，以及不希望使用訂閱制服務的使用者。

---

## 決策

採用**本地優先**架構：

- 所有資料存在使用者本機（IndexedDB via Dexie.js）
- 不設計任何後端伺服器或雲端同步
- 功能 100% 離線可用（PDF 匯出、搜尋、知識圖譜均在前端執行）
- 唯一涉及網路的功能是 `getLinkPreview`（在主程序 `net.fetch` 執行，可選，失敗不影響核心功能）

---

## 後果

### 正面

- **隱私**：使用者資料不離開本機，不存在資料洩漏或第三方存取風險
- **離線**：App 在無網路環境下完全可用（記事、白板、搜尋、匯出均不需網路）
- **免費**：不需後端基礎設施，沒有伺服器成本，因此核心功能可永久免費
- **速度**：讀寫本機 IndexedDB 比網路請求快一個數量級，搜尋無延遲
- **簡單性**：不需要實作認證、授權、衝突解決、同步佇列等複雜邏輯

### 負面

- **無跨裝置同步**：使用者需要自行備份（手動匯出 JSON 或依賴 Backup Panel 的本機備份）
- **資料遺失風險**：磁碟損壞或誤刪 IndexedDB 會導致資料遺失（緩解：30 份自動備份）
- **協作限制**：多人共享同一白板需要手動匯出/匯入，無法即時協作
- **設備更換困難**：更換電腦時需要手動匯出所有白板，體驗不如雲端工具流暢

### 引入的設計約束

- 備份策略（30 份、5 分鐘節流）是本地優先的必要補償機制
- 軟刪除 14 天緩衝是「不可撤銷操作的本地緩衝」設計
- `window.electronAPI` 的 `saveDocument` / `loadDocument` 是歷史殘留的持久化路徑（electron-store），主要持久化已改由 Dexie 承擔
- 匯出功能（JSON/PNG/PDF/Markdown）是使用者在「本地優先沒有雲端備份」場景下的自救手段，因此優先實作

---

## 本地優先的設計原則（在此 App 中的體現）

| 原則 | 實作方式 |
|------|---------|
| 資料完全屬於使用者 | IndexedDB 在本機，不上傳任何資料 |
| 離線優先 | 所有功能（含 PDF 匯出）均在 Electron 前端執行 |
| 可匯出 | JSON / PNG / PDF / Markdown 四種格式匯出 |
| 容錯 | 30 份自動備份 + 14 天軟刪除緩衝 |
| 快速 | 搜尋全量掃描記憶體，無網路延遲 |

---

## 未來演進方向（待確認）

若未來需要加入同步功能，本地優先架構的演進路徑：

1. **自託管同步**：使用 CRDTs（如 Automerge、Yjs）讓多裝置在 P2P 或自己的伺服器上同步，不需中央雲端
2. **選擇性雲端備份**：允許使用者選擇將備份上傳到指定的 S3/Dropbox/本機 WebDAV，而非強制雲端
3. **保持本地核心**：即使加入同步，本機 IndexedDB 仍是「ground truth」，同步是可選的附加層

不建議的方向：
- 強制雲端帳號（破壞本地優先承諾）
- 中央化資料儲存（引入資料所有權問題）
