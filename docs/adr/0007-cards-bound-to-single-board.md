# ADR 0007：接受「卡片綁定單一白板」，以 `[[連結]]` 作為跨板可達的近似解

## 狀態

已採用（2026-07-15）—— 補充 [0002 選用 tldraw](0002-use-tldraw.md) 的一項未言明後果。

---

## 背景

使用者回報的核心痛點：

> 白板一多不好追蹤，復盤筆記也是，需要憶起。

初期把它當成 UI 問題（入口不夠、面板不好用）。但盤點後發現**入口從來不是瓶頸**：卡片庫有 5 個入口、搜尋有 4 個，全部都要使用者「先想起來有東西可找」。

真正的成因在資料模型。

### 本專案的模型

卡片是 tldraw 的 shape，存在**某一個** `BoardRecord.snapshot` 內。一張卡片只能屬於一個白板；要它出現在別處，只能複製（產生副本，兩份各自漂移）或移動（原處消失）。

因此「這張卡在哪個板？」是一個**真實存在的問題**。白板數量一多，使用者靠位置記憶檢索的能力就崩潰——而 App 目前所有的檢索手段（總覽、卡片庫、搜尋、快速切換）都建立在使用者記得「有這個東西」的前提上。

### 對照 Heptabase

Heptabase 的 Card Library 說明：

> 因為每張卡片獨立存在於 card library，同一張卡可以出現在多個白板的不同概念排列中，而不會產生重複。

即：**卡片是獨立實體，白板只是「視角」**。同一張卡可同時出現在多個白板。在該模型下，「這張卡在哪個板」這個問題不存在——卡片在 library 裡，白板只是它的展示場合。

Milanote 則是另一端：刻意不做結構化，靠巢狀板與少量型別，接受「板就是位置」。

---

## 決策

**接受此限制，不重寫資料模型。** 改以 `[[卡片名]]` 雙向連結作為「跨板可達」的低成本近似解，並優先投資讓連結更容易長出來的功能。

理由：

1. **重寫成本與風險極高**。要讓卡片脫離 snapshot 成為獨立 entity，等於重做整個 tldraw 整合（shape 的 ground truth 不再是 tldraw store）、重做 snapshot 讀寫／sanitize／備份／垃圾桶／匯出。規模大於 N18（大型 vault 效能模式），而 N18 本身已被列為「架構級、長期」。
2. **違反既有原則**。專案定位為自用工具，原則是「核心達成即可，別為數字目標硬做高風險低回報重構」。
3. **已有可行的近似解**。`[[連結]]` 讓卡片以**內容關聯**取代**位置關聯**被找到，繞過位置綁定，且基礎設施（`useBacklinks` 增量快取、`extractCardName`、知識圖譜）皆已存在。

---

## 後果

### 正面

- 不動 tldraw 整合，零風險
- 連結是漸進的：用得越多，跨板可達性越好，不需要一次到位
- 明確了投資方向——**該做的是「讓連結更容易長出來」，而不是「再加一個瀏覽介面」**

### 負面 / 取捨

- **位置綁定的痛點不會消失**，只會被連結稀釋。連結沒長到的地方，該卡片仍然只能靠位置找。
- **連結需要使用者主動維護**。這是 App 必須補上的缺口：目前對 `[[]]` 的支援全是被動的（手動打、自動補全、卡內 backlinks 面板、知識圖譜），**沒有任何機制主動告訴使用者「這裡該連」**。
- **孤島卡片偵測短期不可用**。使用者 2026-07 才開始用 `[[]]`，此時「沒有 backlink 的卡片」≈ 全部舊卡，是雜訊而非訊號。連結圖長密後可重新評估。

### 由本 ADR 導出的優先項

- **未連結提及偵測（原 N6，曾併入 A6 而未做）優先序上調**。它是「用新連結打撈舊卡」的機制：寫下 `[[專案A]]` 時，指出「另有 3 張舊卡提到過『專案A』但沒連結」。這同時解決連結維護缺口與「憶起」需求，且自帶時機（在寫作當下浮現，不需使用者記得去看）。
  - **可行性前提**：需驗證能否掛在 `useBacklinks` 的增量快取上。全量掃描是 O(n×m)，而真實 vault 為 445M；`useBacklinks` 本身就曾是效能問題（TD4）後改增量才解決。做之前先驗證，不可揮手帶過。

---

## 未來若要重新評估

出現以下情況可重開此決策：

- 連結圖已足夠密，但「東西在哪個板」仍是主要痛點（代表近似解不夠）
- tldraw 官方提供 shape 跨 document 共享的機制（消除重寫成本）
- 專案定位從自用工具改變
