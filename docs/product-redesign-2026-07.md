## 一、功能全景盤點

盤點目的：重設計前先確認「這個 App 到底有什麼」。

| 分類 | 功能 |

## 二、盤點發現的事實（皆已對照程式碼查證）

### 2.1 整條「死線視角」產品線是死的

使用者用 todo 卡，但**很少設到期日**。以下全部需要 `dueDate` 才有值：

| 位置 | 依據 | 實況 |
|------|------|------|
| 儀表板「逾期任務」 | `t.dueDate < today` | 恆 0 |
| 儀表板「今日到期」 | `t.dueDate === today` | 恆 0 |
| 儀表板「本週完成」 | `Dashboard.tsx:90` `t.dueDate >= weekStart` | 恆 0 |
| 儀表板「今日待辦」widget | `if (t.checked \|\| !t.dueDate) continue` | 恆空 |
| 側邊欄逾期／今日 badge | App.tsx 同算法 | 不出現 |
| 逾期提示 banner | `overdueCount === 0` 即 return | **永不出現** |
| 任務中心 5 頁籤 | 逾期／今日／本週 | 3 個恆空 |

「本週完成」算的是「到期日落在本週的已完成項」，而非「本週勾掉的」——資料模型沒有完成時間欄位。

**儀表板 4 格統計有 3 格恆為 0、主打 widget 恆為空。** 這不是「使用率低」，是介面在展示虛假的空白；兩個中心感覺無用，很可能是此現象的下游。

### 2.2 「久未造訪」的浮現機制已經存在，但幾乎不可見

`BoardTabBar.tsx:303` 已有 `isStale`（`lastVisitedAt` 超過 14 天），在白板名旁顯示 🕐——但樣式是 `fontSize: 9, opacity: 0.4`。**訊號已經算出來了，只是被設計成看不見。**

同理 `Dashboard.tsx:134` 已在用 `lastVisitedAt` 排序，但**方向相反**：它告訴你「剛剛看過什麼」，而使用者要的是「忘了什麼」。

### 2.3 週回顧不需要 dueDate，且資料一定有

`WeeklyReview.tsx:16` `computeWeekStats` 以 `board.updatedAt` 落在本週篩選，再數卡片數、已完成 todo 數、`[[連結]]` 數。**完全不依賴 dueDate。**

週回顧是唯一「數字一定跑得出來」的復盤功能，回答「我這禮拜幹了啥」＝一種憶起。它沒被用**不是因為沒東西可看，是因為躲在一個不會被開啟的中心的第三個頁籤裡，且要求使用者每週主動想起來去看**。

> ⚠️ 修正記錄：討論中途曾據「使用者沒在用」推論「週回顧沒價值、應砍、AI-4 取消」。查證後推翻——**沒被用的原因是位置錯，不是內容沒價值**。

### 2.4 入口重複度極高

卡片庫有 5 個入口（側邊欄／Ctrl+Shift+L／命令面板／儀表板捷徑卡／快捷鍵面板），搜尋有 4 個。**東西找不到的原因從來不是入口太少**，再加入口無效。

## 三、問題的兩次重新框定

### 第一次：D7 的前提是錯的

D7 把任務中心與復盤中心綁在一起討論，但兩者不對稱：

- **復盤中心是個 127 行的殼**，包住三個獨立元件（`CalendarContent`／`JournalDayContent`／`WeeklyReviewContent`），幾乎沒有自己的邏輯。解散它不減任何功能。
- **任務中心有 330 行真邏輯**，跨白板收集 todo 是儀表板給不了的。該留，但要脫掉行事曆外衣。

**更根本的診斷：這兩個中心是按「功能類別」歸檔的，不是按「使用時機」。**

復盤中心三個頁籤（月曆／日記／週回顧）唯一共通點是「都跟時間有關」——那是檔案櫃分類法，不是使用情境。使用者不會有「我現在想做復盤類的事」這種念頭。

對照組：**儀表板是按時機組織的**（「早安，今天是⋯」→ 你現在該知道什麼），所以它是唯一被使用的。Inbox Triage 也是（一次一張、做一個決定）——那是個動作，不是個櫃子。

**兩個中心不是做得不好，是被歸檔在一個使用者腦中不存在的分類下。補丁救不了這個。**

### 第二次：真正的需求是「憶起」，不是死線管理

使用者原話：

> 主要是白板一多不好追蹤，復盤筆記也是，需要憶起。

「不好追蹤」是兩種不同失敗，使用者痛的只有第二種：

- **找不到**（知道有這東西但要翻半天）→ 已有解：Ctrl+P／Ctrl+K／搜尋／卡片庫
- **想不起來**（根本忘了它存在）→ **無解**，所有現有入口都要使用者先想起來

→ 需要的不是「更好的瀏覽介面」，而是**系統主動端東西到面前**。此需求的架構根源見 **[ADR 0007](adr/0007-cards-bound-to-single-board.md)**。

## 四、競品對照：三種對「時間」的答案

使用者反饋「月曆不常使用，因為不知道具體的用途」，故參照 Heptabase／Milanote／Notion。

| 產品 | 日曆的定位 | 對本專案的意涵 |
|------|-----------|---------------|
| **Notion** | Calendar view **必須有 date property**；日曆是結構化資料的鏡頭，背後永遠有使用者明確設定的日期 | 使用者不設 dueDate → **Notion 式日曆在定義上就不成立**，不是做得不好 |
| **Milanote** | **沒有日曆**。只有 checklist 與簡單任務卡；要結構化日期／狀態就叫你去用 Notion | 「不知道用途」就是該砍的信號（呼應少型別哲學） |
| **Heptabase** | 有日曆，但**不是面板**：白板空白處右鍵 → Calendar → 把某段時間的 **journal 卡**以日曆版面排到白板上 | 日曆是「在白板上排列卡片的版面」，排的是使用者親手寫的東西，不是系統時間戳 |

**Heptabase 的 Journal 定位**：「幫你快速捕捉飛逝的想法，不用煩惱要放哪」→ 選取文字 → "Turn into new card" → 拖到白板。

**這正是本專案已有的流程**：快速捕捉 → 收件匣 → Inbox Triage → 移到白板。Heptabase 的核心 capture-then-synthesize，N2（2026-07-15）剛完成。

### 我們的月曆為何無用（`CalendarView.tsx:20-70`）

三個資料源：

| 資料源 | 語意 | 實況 |
|--------|------|------|
| `hasJournal` | 那天有沒有寫日記 | **有意義**（使用者意圖） |
| `todos` where `dueDate` | 那天到期的待辦 | **恆空** |
| `activeBoards` where `board.updatedAt` 落在該日 | 那天「活躍」的白板 | **語意是壞的** |

第三項是關鍵：`updatedAt` 是**單一時間戳＝最後修改時間**，因此每張白板在整個月曆上**只會出現在一天**（最後被碰的那天）。連改一週的白板只出現在最後一天。

**這不是「每日活動記錄」，是「白板最後修改日的散佈圖」——沒有可理解的語意。** 使用者說不出它的用途，是因為它真的沒有。

三個參考裡沒有一個的日曆長得像我們的。

### 重設計後的分層

- **主動層＝儀表板**（唯一主頁，每天必看）：收件匣待整理、本週動態、日記時間軸、未連結提及、未完成任務摘要、最近使用
- **動作層**：Inbox Triage（已完成）
- **視角層**（Ctrl+K 叫出，側邊欄不常駐）：跨白板任務清單、知識圖譜、卡片庫、搜尋
- **寫作層**：白板本身（含 Journal 板）

### 待決／風險

- **AI-4（週回顧 AI 草稿）定位待重估**：若本週動態自動出現在儀表板，可能根本不需要 AI 代寫草稿。
- ~~**未連結提及效能未驗證**~~ → **✅ 已驗，前置解除（2026-07-15）**。效能不是問題，見
  [n6-performance-2026-07-15.md](maintenance/n6-performance-2026-07-15.md)。**真正的風險換成「準確度／雜訊」**：
  樸素子字串比對會把「主頁白板」誤判進含有「主頁白板 (2)」的內文（`uniqueName` 撞名加序號 → 真實 vault 就有這組），
  且 dogfooding 已知句子型標題全數落空。**下一個該解的是比對規則，不是速度。**
- **孤島卡片偵測短期不可用**：使用者 2026-07 才開始用 `[[]]`，此時「無 backlink」≈ 全部舊卡＝雜訊。連結圖長密後再評估。
- **實作順序未定**：死統計清理（低風險、獨立成立）可先做；砍月曆／復盤中心與儀表板重組是較大重構。
