# Scout Astrolabe 檔案與產品功能分析

## 1) 專案定位與技術結構

Scout Astrolabe 是一個以 **Windows 桌面離線知識管理/白板工作流** 為核心的產品，技術上採用：

- **Electron**：打包成桌面應用，提供原生視窗與本機 API。
- **React + Vite + TypeScript**：前端 UI 與功能模組化開發。
- **tldraw**：白板與卡片拖拉互動核心。
- **Dexie (IndexedDB)**：本機資料儲存、快照與備份。

此架構讓產品偏向「本機優先（local-first）」而非雲端協作。

## 2) 關鍵檔案解讀（高層）

- `README.md`：完整描述功能面（卡片類型、任務中心、知識圖譜、回顧中心、備份還原、匯出等）與使用流程。
- `package.json`：可看出產品是 Electron 桌面應用，並具備 Windows 打包流程 `build:win`。
- `src/App.tsx`：主組合層，集中管理各面板（Search / Task / Filter / Review / Backup / Graph / QuickCapture）開關與快捷鍵。
- `src/hooks/useBoardManager.ts`：核心業務邏輯，含白板 CRUD、父子板導航、切換自動備份、Journal/週回顧跳轉、收件匣流程等。
- `src/hooks/useFileStorage.ts`（與相關 utils/db）：支撐本機持久化、資料匯入匯出等能力（local-first 關鍵）。

## 3) 產品功能優勢

1. **功能覆蓋廣，接近一站式知識工作台**  
   從捕捉（Quick Capture）→ 整理（白板/卡片）→ 執行（Todo + Task Center）→ 回顧（Weekly Review）形成閉環，減少工具切換成本。

2. **本機離線與隱私友善**  
   不依賴雲端帳號即可使用，適合重視隱私、弱網環境、或內部知識不外流情境。

3. **白板互動 + 卡片語義兼具**  
   有 tldraw 的視覺化排列優勢，也有 `[[雙向連結]]`、Backlinks、Knowledge Graph 等知識網路能力，兼顧「空間思考」與「語義關聯」。

4. **工作流細節完整**  
   包含逾期提醒、收件匣、快捷鍵、右鍵選單、跨白板移動、主題切換、備份還原，代表對日常使用痛點有實作深度。

5. **技術選型務實，擴充潛力好**  
   React hook 分層 + utils 抽離，後續可持續演進（例如新增同步層、插件系統、卡片模板）。

## 4) 產品功能劣勢與風險

1. **單機導向，協作能力不足**  
   目前聚焦本機儲存，若使用者有多人即時協作、跨裝置同步需求，產品競爭力會弱於雲端筆記/白板工具。

2. **功能面多，學習曲線提高**  
   卡片類型、中心面板、快捷鍵很多，對新手可能造成認知負擔；需更強的引導、模板與預設工作流。

3. **狀態管理逐步變複雜**  
   `App.tsx` 聚合大量 UI 狀態，`useBoardManager` 承載許多業務責任。功能持續增長時，可能面臨可維護性與測試難度上升。

4. **資料可靠性仍需工程化保障**  
   雖有自動備份，但若未建立更完整的版本化、衝突恢復、災難恢復策略，重度用戶仍可能擔心資料安全邊界。

5. **平台覆蓋與生態限制**  
   README 以 Windows 為主，若要擴大用戶，仍需更明確的跨平台策略（macOS/Linux）與分發維運流程。

## 5) 綜合評估

Scout Astrolabe 的定位明確：**離線優先的視覺化知識管理桌面工具**。其最大優勢在於把白板、任務、日記、回顧、知識連結放進同一工作流；最大短板則是缺少雲端協作與跨端同步。若產品下一步走向是個人深度使用者，應優先強化穩定性與可維護性；若要擴張市場，則需補齊同步與協作能力。

## 6) 競品對比（Milanote / Obsidian Canvas / Heptabase）

| 維度 | Scout Astrolabe | Milanote | Obsidian Canvas | Heptabase |
|---|---|---|---|---|
| 核心定位 | 本機離線白板知識工作台 | 雲端視覺化專案與創意整理 | 本機筆記延伸的白板視圖 | 視覺化研究與知識建構 |
| 主要優勢 | 離線、隱私、任務+回顧+圖譜整合 | 協作成熟、上手直覺、分享便利 | 生態插件強、Markdown 工作流成熟 | 卡片思考與知識串接體驗強 |
| 主要短板 | 協作/同步較弱 | 進階知識連結與本機控制有限 | 視覺化互動與專案管理整合度較分散 | 對新手與輕量使用者較重 |
| 適合族群 | 重視本機資料掌控的個人/小團隊 | 跨部門協作、設計/內容團隊 | 已深度使用 Obsidian 的知識工作者 | 研究者、長期知識工程使用者 |

### Scout 相對競品的機會點

1. **以「離線 + 任務執行 + 回顧」形成差異化**：
   不是只做白板，而是把週期性產出流程（Capture→Plan→Do→Review）產品化。

2. **補上「可選擇的同步層」**：
   保留 local-first，另提供可關閉的雲同步/局域網同步，兼顧隱私與跨裝置便利性。

3. **建立模板與情境化入口**：
   提供研究、產品規劃、內容企劃、個人 GTD 模板，降低學習成本並提升留存。

## 7) 優先開發路線圖（30/60/90 天）

> 目標：先提升穩定性與日常可用性，再逐步擴張成長能力。

### Phase 1（0–30 天）：穩定性與可維護性

- **P0：資料安全強化**
  - 備份中心增加「手動快照命名」與「一鍵回滾最近版本」。
  - 匯出備份附帶 metadata 版本號，降低未來 schema 變更風險。
- **P0：狀態管理拆分**
  - 將 `App.tsx` 面板開關與快捷鍵邏輯拆成獨立 hooks（如 `usePanels`, `useGlobalHotkeys`）。
  - 將 `useBoardManager` 中 Journal/導航/備份責任分層。
- **P1：可觀測性**
  - 新增錯誤日誌與操作事件（本機）以便排查資料損毀與功能失效。

**成功指標（KPI）**
- 備份還原成功率 ≥ 99%。
- 主要白板切換/儲存錯誤回報下降（相較前版本）。

### Phase 2（31–60 天）：體驗與效率

- **P0：新手導覽 2.0**
  - 以任務清單式 onboarding 引導完成：建卡、連結、收件匣整理、週回顧。
- **P0：模板系統（v1）**
  - 預設模板：週計畫、研究筆記、內容日曆、產品需求拆解。
- **P1：搜尋與篩選升級**
  - 支援複合查詢（tag + status + 日期）與近期操作排序。

**成功指標（KPI）**
- 首次使用 24 小時內完成「建立 3 張卡 + 1 次回顧」比例提升。
- 搜尋到目標卡片平均操作步數下降。

### Phase 3（61–90 天）：成長與擴張

- **P0：可選同步方案（Beta）**
  - 先做「裝置間備份包同步」或「雲端檔案夾同步」的低風險版本。
- **P1：跨平台驗證**
  - 完成 macOS 內測包（功能 parity 先達 80%）。
- **P1：分享能力（非即時協作）**
  - 支援只讀快照匯出（Markdown + 圖片資產）供團隊傳閱。

**成功指標（KPI）**
- 7 日留存提升。
- 多裝置使用者比例提升。
- 匯出/分享功能使用率提升。

---

### 建議執行順序（簡版）

1. 先把「資料安全」和「程式結構可維護」打穩。
2. 再把「上手門檻」降下來（導覽 + 模板）。
3. 最後引入「同步/跨平台」作為成長引擎。

## 8) 題外話：這時候加入 AI 會比較好嗎？

結論：**可以加入，但建議採「小步快跑、以輔助為主」策略**。  
以目前產品型態（local-first、白板卡片、任務與回顧）來看，AI 最適合先做「提高整理效率」而非「取代核心編輯」。

### 什麼情況下「現在就加 AI」是加分

1. 使用者已有大量卡片，但整理成本高（找不到重點、難以連結）。
2. 使用者有固定週期工作流（每日記錄、每週回顧、任務拆解），適合 AI 做摘要與重組。
3. 你可以先接受「半自動」體驗（AI 提案，使用者確認）而不是全自動決策。

### 不建議一開始就重壓 AI 的情況

1. 核心穩定性（儲存、備份、還原）尚未完全打穩。
2. 產品基礎體驗仍在快速變動（資訊架構、卡片模型、快捷流）。
3. 還沒有定義清楚 AI 成功指標（例如節省多少時間、提升多少留存）。

## 9) AI 功能優先順序（建議）

### 第一層：低風險、高價值（建議先做）

- **卡片摘要（Summarize）**：長文本/會議筆記一鍵摘要成 3–5 點。
- **待辦拆解（Task Breakdown）**：把一段目標文字拆成可執行 todo。
- **週回顧草稿（Weekly Draft）**：依本週卡片自動生成「完成/卡住/下週目標」。
- **連結建議（Link Suggestions）**：建議可互相連結的卡片，使用者一鍵接受。

### 第二層：中風險、中高價值

- **語意搜尋（Semantic Search）**：不用精準關鍵字也能找到相關卡片。
- **白板重整建議（Board Refactor）**：建議聚類、標籤、優先級。

### 第三層：高風險、高成本（後期）

- **自主代理操作（Agentic Actions）**：AI 自動新增/移動/刪除卡片。
- **跨板長鏈推理（Cross-board reasoning）**：自動產出策略建議或計畫藍圖。

## 10) 導入 AI 必須注意的 8 件事

1. **隱私與資料邊界（最優先）**  
   明確區分：哪些資料可送雲端模型、哪些只能留本機。提供「AI 完全關閉」選項。

2. **人機權責設計**  
   AI 只「建議」不直接覆寫；所有寫入操作都要可撤銷（Undo）與可追溯。

3. **可解釋性與來源透明**  
   摘要/建議應標註來源卡片，避免使用者不知道 AI 依據什麼產生內容。

4. **成本控制**  
   對 token、模型層級、請求頻率做預算上限，避免每月成本失控。

5. **延遲與離線回退**  
   設計 loading 與 timeout；網路差時要有降級模式（例如改成本機規則引擎）。

6. **品質評估機制**  
   為每個 AI 功能定義可量化 KPI：採納率、節省時間、誤建議率。

7. **Prompt 與版本治理**  
   把 prompt 當作可版本化資產管理，避免每次修改都不可回歸比較。

8. **法規與合規**  
   若涉及企業客戶，需提前準備資料處理聲明、模型供應商條款、區域合規策略。

## 11) 你可以直接採用的落地方案（90 天內）

- **第 1–30 天**：先上「摘要 + 週回顧草稿 + 待辦拆解」三件套（僅建議模式）。
- **第 31–60 天**：加入連結建議與 AI 採納率儀表板。
- **第 61–90 天**：評估語意搜尋 Beta，並建立成本/品質雙指標決策門檻。

**決策原則（簡短版）**：  
先把 AI 當「效率加速器」，不要當「自動駕駛」。在 local-first 產品中，信任與可控性比炫技更重要。

## 12) 總結報告（Executive Summary）

### A. 產品現況一句話

Scout Astrolabe 已具備「離線白板 + 任務執行 + 知識連結 + 週期回顧」的完整個人知識工作流雛形，差異化清楚，但需補強可維護性與成長能力。

### B. 目前最關鍵的 3 個優勢

1. **流程閉環完整**：Capture → Organize → Execute → Review 已可在同一工具內完成。  
2. **Local-first 信任優勢**：離線可用、隱私友善，適合個人與小團隊深度使用。  
3. **視覺與語義並重**：白板空間思考 + 雙向連結/圖譜，兼顧發想與知識沉澱。

### C. 目前最關鍵的 3 個風險

1. **同步與協作能力不足**：擴張到多人/跨裝置場景會受限。  
2. **程式複雜度正在上升**：`App.tsx` 與 `useBoardManager` 責任偏重，後續維護成本可能提高。  
3. **資料治理尚需工程化**：備份雖存在，但版本化、恢復策略與可觀測性仍需補齊。

### D. 接下來 90 天建議（管理層版本）

- **第 0–30 天（打底）**：先做資料安全與結構重整（備份/還原、狀態拆分、可觀測性）。
- **第 31–60 天（提效）**：上新手導覽 2.0、模板系統、搜尋篩選增強，降低上手成本。
- **第 61–90 天（成長）**：試行可選同步 Beta、跨平台驗證、只讀分享輸出。

### E. AI 導入決策（超精簡）

- **要做**：先做「摘要、待辦拆解、週回顧草稿」等輔助型 AI。  
- **不要急著做**：高自主代理（自動改資料）與高風險長鏈決策。  
- **原則**：人機協作、可撤銷、可追溯、可關閉、可控成本。

### F. 建議追蹤 KPI

- 產品穩定：備份還原成功率、白板切換儲存錯誤率。  
- 使用效率：搜尋步數、任務完成率、週回顧使用率。  
- 成長指標：7 日留存、多裝置使用比例、匯出/分享使用率。  
- AI 成效：AI 建議採納率、單次任務完成時間下降比例、誤建議率。

### G. 結論

這個產品方向是對的。短期優先順序應該是：
**先穩定（資料與架構）→ 再提效（導覽與模板）→ 後成長（同步與 AI）**。  
只要維持 local-first 信任優勢，同時逐步補上同步與 AI 輔助能力，Scout Astrolabe 有機會在個人知識工作台市場建立明確定位。
